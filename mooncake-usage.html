E2B Mooncake Storage 使用说明

本文说明如何在 e2b-infra 中使用 Mooncake 作为 storage backend，以及如何在具备 Mooncake C/Go binding 和 URMA transport 的节点上启用 host cache 注册，减少 Mooncake 到本机 mmap cache 的中间拷贝。

默认模式：只需要 HTTP Mooncake 服务，使用 STORAGE_PROVIDER=Mooncake 和 MOONCAKE_HTTP_ENDPOINT。

零拷贝模式：需要用 -tags mooncake_cgo 编译，并设置 MOONCAKE_NATIVE_ENABLED=true。chunker 创建本地 mmap cache 后会调用 Mooncake native client 的 RegisterBuffer，读取 chunk 时使用 GetInto 直接写入已注册的 mmap slice。

普通 Mooncake HTTP 模式

该模式不依赖 Mooncake C 库或 URMA。它保持现有 HTTP object API 行为，适合开发、CI 和没有 RDMA/URMA 环境的节点。

export STORAGE_PROVIDER=Mooncake
export MOONCAKE_HTTP_ENDPOINT=http://mooncake-http.example:8080
export MOONCAKE_NAMESPACE=templates

# 可选
export MOONCAKE_UPLOAD_CONCURRENCY=16
export MOONCAKE_WRITE_TIMEOUT_SECONDS=30
export MOONCAKE_READ_TIMEOUT_SECONDS=15
export MOONCAKE_OPERATION_TIMEOUT_SECONDS=5

URMA 零拷贝模式

该模式依赖 Mooncake native Go binding 和 Mooncake C/C++ build 产物。不要在普通节点启用 mooncake_cgo tag；只有具备 Mooncake 头文件、动态库、URMA/ibverbs 运行环境的节点才应该使用。

构建前准备

Mooncake Go binding 使用模块路径 github.com/kvcache-ai/Mooncake/mooncake-store/go。如果该模块不在公网模块代理中，请在有 Mooncake 源码的节点上用 workspace 或 replace 指向本地源码。

# 示例：在编译节点创建本地 workspace 映射
go work use /home/lixu/e2b-infra/packages/shared
go work use /home/lixu/e2b-infra/packages/orchestrator
go work use /home/lixu/Mooncake/mooncake-store/go

CGO flags

按 Mooncake build 目录调整以下路径。URMA 环境通常还需要系统库路径能找到 liburma.so、libibverbs.so 等依赖。

export CGO_ENABLED=1
export CGO_CFLAGS="-I/home/lixu/Mooncake/mooncake-store/include -I/home/lixu/Mooncake/mooncake-transfer-engine/include"
export CGO_LDFLAGS="-L/home/lixu/Mooncake/build/mooncake-store/src \
-L/home/lixu/Mooncake/build/mooncake-store/src/cachelib_memory_allocator \
-L/home/lixu/Mooncake/build/mooncake-transfer-engine/src \
-L/home/lixu/Mooncake/build/mooncake-transfer-engine/src/common/base \
-L/home/lixu/Mooncake/build/mooncake-common \
-L/home/lixu/Mooncake/build/mooncake-common/src \
-lmooncake_store -lcachelib_memory_allocator -ltransfer_engine -lbase -lasio -lmooncake_common \
-lstdc++ -lnuma -lglog -lgflags -libverbs -ljsoncpp -lzstd -lcurl -luring"

运行参数

变量用途示例或默认值
STORAGE_PROVIDER选择 Mooncake backendMooncake
MOONCAKE_NATIVE_ENABLED启用 native client 和 buffer registrationtrue
MOONCAKE_NAMESPACE对象 key 前缀；为空时使用 bucket nametemplates
MC_METADATA_SERVERMooncake metadata server 地址http://host:8080/metadata
MC_MASTER_ADDRMooncake master gRPC 地址host:50051
MC_LOCAL_HOSTNAME本节点给 Mooncake transfer engine 使用的 hostname/IPlocalhost
MC_PROTOCOLMooncake transport protocolrdma
MC_DEVICE_NAMERDMA/URMA 设备名；为空时交给 Mooncake 自动发现
MC_GLOBAL_SEGMENT_SIZEMooncake global segment size536870912
MC_LOCAL_BUFFER_SIZEMooncake local transfer buffer size134217728
 
export STORAGE_PROVIDER=Mooncake
export MOONCAKE_NATIVE_ENABLED=true
export MOONCAKE_NAMESPACE=templates
export MC_METADATA_SERVER=http://mooncake-master:8080/metadata
export MC_MASTER_ADDR=mooncake-master:50051
export MC_LOCAL_HOSTNAME=$(hostname -I | awk '{print $1}')
export MC_PROTOCOL=rdma

go test -tags mooncake_cgo ./packages/shared/pkg/storage
go test -tags mooncake_cgo ./packages/orchestrator/pkg/sandbox/block

零拷贝生效路径

template/build diff 读取时，orchestrator 的 chunker 会创建本地 mmap cache。若 upstream storage object 实现了 buffer registration，chunker 初始化阶段注册整段 mmap；每次 fetch chunk 时，如果 upstream 支持 direct range read，就把 Mooncake object 直接读入该 mmap slice。chunker 关闭时先 unregister，再 unmap 和删除 cache 文件。

排查

现象处理
MOONCAKE_NATIVE_ENABLED=true requires building with -tags mooncake_cgo当前 binary 未启用 native build tag，重新用 -tags mooncake_cgo 编译。
fatal error: store_c.h: No such file or directory检查 CGO_CFLAGS 是否包含 mooncake-store/include。
链接 Mooncake 符号失败检查 CGO_LDFLAGS 中 Mooncake build 目录、库顺序和系统依赖。必要时参考 Mooncake 的 mooncake-store/go/build.sh。
注册 buffer 失败确认 URMA/RDMA 设备、权限、memlock limit、hugepage/ibverbs/urma runtime 已正确配置。
