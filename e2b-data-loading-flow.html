E2B 数据加载流程与 Mooncake 性能分析

本文描述 sandbox 启动时 template 数据从持久化存储加载到本机 host cache，再被 Firecracker/Rootfs/NBD/UFFD 读取的完整链路；同时对比传统对象存储路径和 Mooncake 路径，说明 Mooncake native + URMA 注册 host cache 后的性能收益与限制。

核心路径GetTemplate 命中 template cache 后直接复用；未命中时构造 storageTemplate，后台 Fetch 并行准备 snapfile、metadata、memfile、rootfs。

数据面memfile/rootfs 不是一次性全量下载，而是通过 StorageDiff 和 Chunker 按 chunk 懒加载到 mmap cache。

Mooncake zero-copy 当 Mooncake native seekable 同时实现 BufferRegistrar 和 DirectRangeReader 时，chunker 注册整段 mmap cache，并让 Mooncake 直接写入该 mmap slice。

一、E2B 数据对象

对象用途加载特点
metadatatemplate 元数据，描述版本、层、预取等信息。普通 blob 文件，启动早期读取；缺失时可回退 v1 metadata。
snapfileFirecracker snapshot state。普通文件，使用 newStorageFile 缓存到本地路径。
memfileVM 内存镜像或增量 diff。seekable，大文件，按 chunk/page fault 懒加载到 mmap cache。
rootfs根文件系统镜像或增量 diff。seekable，大文件，NBD/overlay 读取触发懒加载。
header描述 build diff 链、block size、压缩 frame table、base build 等。先于 StorageDiff 解析，决定读取链路和解压映射。

二、启动时完整加载流程

1. 请求 templateserver 或 build 流程调用 template.Cache.GetTemplate(buildID)。
2. 查本机 TTL cachettlcache 命中则复用已创建的 Template 和 diff store。
3. 构造 storageTemplate根据 buildID 生成 remote path 和 local cache path。
4. 后台 Fetch并行准备 snapfile、metadata、memfile、rootfs。
5. 加载 headerNewStorage 获取 memfile/rootfs header；老格式会回退推导 header。
6. 创建 build.File按 header 中的 build chain 决定从哪个 diff 读取。
7. 初始化 StorageDiffOpenSeekable 打开远端对象，创建本地 chunker mmap cache。
8. 懒加载数据首次读某个范围时 chunker fetch 远端 chunk，写入 mmap 并标记 cached。

三、传统数据加载方式

传统模式包括 GCS、S3、本地 filesystem，或在其上包一层 NFS cache / peer routing。核心特征是远端对象通过 stream reader 进入进程，再由 Go 代码复制到 chunker 的 mmap cache。

传统 seekable read 路径

Sandbox start
  -> GetTemplate(buildID)
  -> storageTemplate.Fetch()
  -> NewStorage(memfile/rootfs)
  -> build.File.getBuild()
  -> StorageDiff.Init()
  -> persistence.OpenSeekable(path)
  -> block.NewChunker(size, blockSize, seekable, localCachePath)
  -> first ReadAt/Slice(page range)
  -> Chunker.runFetch()
  -> seekable.OpenRangeReader(offset, length)
  -> io.ReadFull(reader, mmapSlice[...])
  -> mark chunk cached

传统路径的数据拷贝

环节典型行为开销来源
网络协议栈HTTP/TLS/GCS/S3 response body 流式读取。内核 buffer、TLS 解密、HTTP framing、用户态 reader。
Go reader 到 mmapio.ReadFull 把数据写入 mmapSlice。用户态复制、批大小控制、goroutine 调度。
压缩对象按 frame 定位压缩范围，读取后解压为 uncompressed chunk。CPU 解压和 frame 边界查找；压缩可减少网络流量但增加 CPU。
本机 cache 命中后续读取直接返回 mmap slice。主要剩下内存访问和 bitmap/cache 状态检查。

四、基于 Mooncake 的数据加载方式

Mooncake backend 有两种形态：HTTP 兼容模式和 native zero-copy 模式。HTTP 模式仍是传统 stream/copy 路径；native 模式通过 Mooncake Store Go binding 接入 RegisterBuffer 和 GetInto，使远端数据直接落到 chunker mmap cache。

Mooncake HTTP 模式

兼容优先无 cgo

配置 STORAGE_PROVIDER=Mooncake 与 MOONCAKE_HTTP_ENDPOINT 后，storage_mooncake.go 使用 HTTP object API。它可以复用现有 seekable/block manifest 逻辑，但仍存在 HTTP body 到 mmap cache 的用户态拷贝。

Mooncake native + URMA

mooncake_cgoRegisterBufferGetInto

启用 MOONCAKE_NATIVE_ENABLED=true 后，native client 在有 Mooncake 库的节点连接 metadata/master。chunker 创建 mmap 后注册整段 host cache，读取时调用 ReadRangeInto，最终使用 Mooncake GetInto 写入已注册地址。

Mooncake native zero-copy read 路径

StorageDiff.Init()
  -> persistence.OpenSeekable(path) returns mooncakeSeekable(native client)
  -> block.NewChunker(...)
  -> cache.RegisterWith(upstream)
       -> nativeMooncakeClient.RegisterBuffer(mmapPtr, mmapSize)
  -> first ReadAt/Slice(page range)
  -> Chunker.runFetch()
  -> upstream.ReadRangeInto(mmapSlice, chunkOffset)
       -> mooncakeSeekable.ReadRangeInto(...)
       -> nativeMooncakeClient.ReadIntoRegistered(...)
       -> Mooncake Store GetInto(key, mmapPtr+offset, len)
  -> mark chunk cached
  -> Chunker.Close()
       -> UnregisterBuffer(mmapPtr)
       -> mmap.Unmap()
这里的“零拷贝”指 Mooncake/URMA 到 e2b host mmap cache 的数据面不再经过 Go reader 中间 buffer。VM 侧读取、NBD/UFFD、CPU page table、存储设备或网络栈内部仍可能存在硬件/内核层面的 DMA、cache coherency 和页表管理开销。

五、性能分析对比

维度传统对象存储Mooncake HTTPMooncake native + URMA zero-copy
远端访问协议GCS/S3 HTTP range 或类似对象存储 API。Mooncake HTTP object API。Mooncake Store client + RDMA/URMA transfer engine。
落入 host cachereader -> Go buffer/read loop -> mmap slice。HTTP response -> read loop -> mmap slice。Mooncake GetInto 直接写入已注册 mmap slice。
CPU 消耗较高，HTTP/TLS、copy、可能解压。中等，后端更近时网络开销下降，但 HTTP/copy 仍在。最低，减少用户态 copy 和协议处理，CPU 更接近调度/metadata/bitmap 开销。
首 chunk 延迟受对象存储 RTT、range request、TLS 和跨 AZ/公网路径影响。取决于 Mooncake 集群距离和 HTTP 服务实现，通常低于云对象存储。主要取决于 Mooncake metadata 查找、RDMA path 和远端 segment 状态，预期最低。
吞吐上限常受 HTTP 并发、对象存储限速、CPU copy 限制。后端更可控，但 HTTP handler 和 copy 仍可能成为瓶颈。更接近网卡/RDMA fabric、Mooncake segment 和内存带宽上限。
本机重复读取命中 mmap cache 后很快。同传统路径。同传统路径；优势主要在首次远端 fetch。
运维复杂度最低，依赖云存储或文件系统。中等，需要 Mooncake HTTP 服务。最高，需要 Mooncake C/C++ 库、Go binding、URMA/RDMA 设备、memlock、库路径和 build tag。
失败回退成熟稳定。可作为 native 不启用时的兼容路径。注册失败会使 chunker 初始化失败，应通过部署策略控制只在合格节点启用。

定性性能结论

场景主要瓶颈最佳路径原因
冷启动首次加载大 memfile/rootfs远端读取、用户态 copy、对象存储 RTT。Mooncake native + URMA减少协议栈和 copy，数据直接进入 host mmap cache。
同一节点重复启动相同 template本机 cache 命中后的内存访问。差异较小chunk 已在 mmap cache 中，远端 backend 不再参与。
跨节点复用热 template跨节点远端拉取和热点带宽。Mooncake native 或 peer routingMooncake 可作为分布式低延迟数据面；peer routing 可减少对象存储回源。
压缩 diff解压 CPU 和 frame 读取。视数据而定压缩降低网络流量，但 zero-copy 对解压后的 CPU 开销帮助有限；当前 Mooncake seekable 对 compressed path 有限制。
小对象 metadata/snapfile请求 RTT 和调度。Mooncake HTTP/native 差距有限对象较小，copy 不是主要开销。

六、性能观测建议

对比时应分开观测冷启动、热启动、首 chunk、全量恢复和 page fault tail latency。推荐至少记录以下指标：

指标含义相关代码/埋点
orchestrator.templates.cache.hits/missestemplate TTL cache 是否命中。template/cache.go
orchestrator.blocks.chunks.fetch远端 chunk fetch 耗时和字节数。block/streaming_chunk.go
orchestrator.blocks.chunks.store写入 mmap cache 的耗时。progressiveRead
orchestrator.blocks.slices读取 slice 的整体耗时，区分 cache hit 和 remote fetch。Chunker.Slice
Mooncake transfer metricsRDMA/URMA 传输吞吐、失败率、segment 状态。Mooncake runtime / monitoring。
